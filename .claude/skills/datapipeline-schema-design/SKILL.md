# Schema Design

데이터 파이프라인의 스키마를 레이어별로 설계하는 방법론. 구현 DDL이 아니라 "어떤 모델이 왜 적합한가"의 설계 문서를 만든다.

## 작성 목표

산출물은 `_workspace/datapipeline/{ts}/01_schema_design.md`. 다른 전문가(ETL·검증·관측성)가 이 문서만으로 자기 영역 설계를 시작할 수 있어야 한다.

## 설계 원칙

- **쿼리 패턴이 스키마를 결정한다** — 주요 쿼리 3~5개를 먼저 상정한다. 쿼리 없는 정규형은 근거 없음.
- **레이어별 책임 분리** — raw는 원본 보존, staging은 정제·표준화, warehouse는 통합 모델, mart는 소비자 최적화. 한 레이어가 모든 책임을 지지 않는다.
- **정규화는 수단, 비정규화는 결과** — warehouse는 3NF 기본. mart는 성능을 위해 의도적 비정규화 가능. 어느 레이어에서 왜 비정규화했는지 명시한다.
- **타입은 보수적으로** — 문자열은 길이 상한 지정(VARCHAR(n)). 금액은 DECIMAL(p,s). 타임스탬프는 UTC + 정밀도 고정(ms/µs). nullable은 의미 있을 때만.
- **파티션은 쿼리 필터와 일치** — 대개 시간(이벤트 일자, 처리 일자). 적정 파티션 크기는 1GB~수십GB, 파티션 수 폭발(> 수만) 방지.
- **스키마 진화는 초기 설계의 일부** — 사후 고민은 늦다. 처음부터 "어떤 변경이 허용/금지인가"를 문서화한다.
- **대안 병기** — 정규 vs 비정규, 파티션 기준 A vs B 같은 선택지를 2~3개 병기하고 선택 기준(비용·지연·복잡도)을 명시한다.

## 워크플로우

### Step 1: 쿼리 패턴 확인

리더 브리프에서 다운스트림 소비자를 확인한다. 불명확하면 SendMessage로 리더에게 질의. 주요 쿼리 3~5개를 가정이라도 기록한다. 예:
- "최근 N일 이벤트 집계 (group by date, dimension)"
- "특정 사용자의 이력 조회 (point query)"
- "월간 코호트 분석 (time-range scan)"

### Step 2: 레이어 맵 그리기

어떤 레이어가 필요한지 결정한다. 기본 권장 구조:

```
source   → (접근 가능한 원본)
 ↓
raw      → 원본 그대로 보존 (JSON/Parquet, 무변환)
 ↓
staging  → 타입 표준화, 스키마 평탄화, 중복 제거
 ↓
warehouse → 통합 모델 (팩트·디멘전 또는 이벤트 + 엔티티)
 ↓
mart     → 소비자별 비정규화 뷰/테이블
```

소규모·단순 파이프라인은 staging 또는 mart 생략 가능. 설계 문서에 각 레이어의 **목적·보존 기간·소비자**를 명시한다.

### Step 3: 엔티티 카탈로그 + ERD

핵심 엔티티 목록을 만들고, 각 엔티티별:
- 목적 (무엇을 나타내는가)
- 입력 소스
- 주요 속성
- 관계 (1:1 / 1:N / N:M)
- 존재 레이어 (raw/staging/warehouse/mart 중 어디에 존재)

Mermaid `erDiagram`으로 ERD를 그린다.

### Step 4: 팩트/디멘전 분류 (분석 계층인 경우)

warehouse가 분석 목적이면 팩트-디멘전 모델을 고려한다:
- **팩트(fact)**: 측정값 + 외래키. 시간에 따라 행이 축적. 예: `fact_order_item`
- **디멘전(dim)**: 기술(descriptive) 속성. 변경 시 SCD 규칙 적용. 예: `dim_customer`, `dim_product`

팩트 테이블은 **그레인(grain)** 을 명시한다. "하나의 행이 무엇을 나타내는가" — 예: "주문 1건의 아이템 1개".

이벤트 중심 파이프라인이면 팩트 = 이벤트 로그, 디멘전 = 엔티티 상태 스냅샷으로 매핑할 수 있다.

### Step 5: SCD(Slowly Changing Dimension) 타입 선정

디멘전에 적용할 SCD 타입:
- **Type 1**: 덮어쓰기. 히스토리 없음. 사용처: 이름 오탈자 수정 같은 교정.
- **Type 2**: 버전 레코드 추가 (valid_from/valid_to + is_current). 사용처: 가격 변경, 주소 이전 등 히스토리 필요.
- **Type 6**: Type 1 + Type 2 혼합 (현재값 컬럼 + 히스토리 레코드). 사용처: 최신 조회 자주 + 히스토리도 필요.

기본 권장: 비즈니스 중요 디멘전은 Type 2. 단순 lookup은 Type 1. 최종 선택은 사용자에게 위임.

### Step 6: 키 전략

엔티티별로 결정:
- **자연키 vs 대리키** — 원본 식별자를 PK로 쓸지(자연키), 합성 ID를 생성할지(대리키). 대리키 기본 권장 (소스 변경·포맷 변화에 독립).
- **복합키 허용 여부** — 팩트 테이블의 그레인 키는 복합 가능.
- **외래키** — warehouse에서는 참조 무결성을 위해 정의, mart는 성능을 위해 생략 가능.
- **파티션 키** — 대개 시간 컬럼(이벤트 일자 또는 ingested_at). 쿼리 대부분이 시간 범위 필터를 쓴다는 전제.
- **클러스터 키 / 분산 키** — 웨어하우스 제품별 명칭 다름. 조인·정렬 자주 쓰이는 키를 지정. 일반 해법 먼저 제시하고 제품별 대응은 부록.

### Step 7: 타입·제약 표

컬럼 단위 표 작성. 필드:
- 컬럼명
- 타입 (일반형 — VARCHAR/DECIMAL/TIMESTAMP/UUID/JSONB)
- 정밀도/길이
- nullable (true/false + 이유)
- 제약 (unique/check)
- 기본값
- 설명

원칙:
- 문자열: 의미 있는 상한 지정. 무제한 TEXT는 청구지 주소 같은 비정형 필드에만.
- 수치: 금액은 DECIMAL(p,s), 측정량은 적절한 범위 검토.
- 타임스탬프: UTC + 정밀도 명시(ms/µs). "로컬 시간" 저장 금지.
- 불린: NULL vs false 구분 필요한지 확인.
- JSON/JSONB: 자주 필터·조인하는 키는 컬럼으로 승격.

### Step 8: 감사 컬럼 표준

모든 레이어 공통으로 추가할 메타 컬럼:
- `ingested_at` — 파이프라인 적재 시각(UTC)
- `source` — 데이터 소스 식별자
- `source_id` — 원본 레코드 식별자
- `batch_id` / `run_id` — 적재 배치 식별
- `content_hash` (선택) — 중복 감지·변경 감지용
- `schema_version` (선택) — 스키마 드리프트 감지

warehouse·mart에는 추가로:
- `created_at`, `updated_at` — 데이터 변경 추적
- SCD Type 2 대상: `valid_from`, `valid_to`, `is_current`

### Step 9: 스키마 진화 정책

다음 정책을 문서화:
- **허용 변경**: nullable 컬럼 추가, 길이 확장, 설명 수정 등 (backward-compatible)
- **제한 변경**: 타입 변경, 제약 강화 — 마이그레이션 절차 필수
- **금지 변경**: PK/파티션 키 변경, 컬럼 삭제 — 새 테이블 + 데이터 이관 절차

스키마 버전 관리 방식: 파일(Liquibase/Flyway/Prisma Migrate), 카탈로그(Schema Registry), 또는 수동. 프로젝트 맥락에 맞게 제안.

### Step 10: 규모 추정 및 읽기 패턴 매칭

각 테이블별:
- 예상 행 수 / 일
- 예상 용량 / 일
- 주요 쿼리 예시 + 어떤 인덱스/파티션을 쓰는가

이 추정이 파티션 선택·비정규화 여부·인덱스 결정의 근거가 된다.

- [ ] ETL 엔지니어와 파티션 키 합의 (SendMessage)

## 금지 사항

- 쿼리 패턴 없이 스키마 작성
- "최선의 스키마" 단일 안 강요 (최소 2개 대안 필수)
- 상용 제품 고유 기능 전제 (일반 해법 먼저)
- 모든 컬럼 nullable 허용
- 감사 컬럼 생략
- 스키마 진화 정책 생략
