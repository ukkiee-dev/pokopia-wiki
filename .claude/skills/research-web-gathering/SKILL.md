# 리서치: 웹 소스 수집

이 스킬은 리서치 주제의 **웹 증거**를 체계적으로 수집하는 방법을 표준화한다. 단순 검색이 아니라 소스 계층·편향·시간 맥락을 명시한 산출물을 만든다.

## 핵심 원칙

1. **쿼리 다변화** — 4~6가지 각도 (직접·반대·공식·시간한정·비교·문제)
2. **1~2차 소스 추적** — 3차 소스(Wikipedia, aggregator)는 단서로만
3. **시간 맥락 명시** — 모든 발견에 발행일 + 접근일
4. **이해관계 공개** — 출처의 편향 가능성 표기
5. **공백 투명성** — 못 찾은 건 "찾지 못함"으로 기록

## 왜 이렇게 엄격한가

리서치 품질은 **소스 품질**에 비례한다. 3차 소스 체인("Wikipedia 말이… 누가 말하기를… 어디서 봤는데…")으로 쌓은 보고서는 원 주장이 왜곡됐을 수 있다. 1차 원문까지 추적하면 왜곡 기회가 줄고, 이해관계를 명시하면 독자가 가중치를 판단할 수 있다.

## 소스 계층

| 계층 | 예 | 신뢰도 | 인용 가능 여부 |
|------|-----|-------|---------------|
| 1차 (원문/발표자) | 공식 프레스릴리스, 정부 보고서, RFC/W3C, 판결문 | 높음 | 최우선 인용 |
| 준1차 (저자 진술) | 당사자 블로그, 컨퍼런스 녹화 | 높음 | 직접 인용 OK |
| 2차 (독립 분석) | NYT/FT/Reuters, 업계 분석지 | 중간~높음 | 인용 OK, 편향 표기 |
| 2차 (블로그·매거진) | TechCrunch, The Verge, Ars | 중간 | 인용 OK, 출처 평판 고려 |
| 3차 (aggregator) | Wikipedia, 요약 사이트 | 낮음 | **단서로만, 원문 추적 후 1~2차 인용** |
| 비공식 | 익명 블로그, SEO 팜 | 낮음 | 인용 금지 (교차 확인 필수) |

## 쿼리 설계 (6가지 템플릿)

주제 `T`에 대해:

```
1. 직접:        "T"
2. 반대 입장:   "T criticism" OR "T failure" OR "T controversy"
3. 공식:        "T" site:{official_domain}
4. 시간 한정:   "T" after:2025
5. 비교:        "T vs alternative"
6. 문제:        "T problem" OR "T issue" OR "T risk"
```

추가 각도:
- 지역 한정: 한국 관련이면 한국어 쿼리 병행
- 업계 한정: `site:.gov`, `site:.edu`, `site:.org`
- 학술성 있는 백서: `filetype:pdf "T"`

## 탐색 루프

```
1. WebSearch(query) → 상위 10개 후보
2. 각 후보 판별:
   ├── 1~2차 소스로 보임 → WebFetch → 핵심 추출
   ├── 3차 소스 → 원문 링크 추적 → 1~2차로 이동
   └── 저품질 → 스킵 + "제외" 사유 기록
3. 다음 쿼리 실행 (총 6회 이상)
4. 중복 소스 발견 → 여러 쿼리에서 같은 소스면 관련성 신호
5. 언어 확장 (한국어 주제 + 영어 검색으로 보완)
```

저장 경로: `_workspace/research/{topic-slug}/web_findings.md`

## 품질 체크리스트 (출력 전)

- [ ] 쿼리 6가지 이상 수행
- [ ] 모든 주요 발견에 URL + 발행일 + 접근일
- [ ] 1~2차 소스 비율 ≥ 70% (3차만 기반이면 부족)
- [ ] 이해관계 공개된 출처에 편향 표기
- [ ] 공백 섹션 존재 (숨기지 않음)
- [ ] 시간순 맥락 정리
- [ ] paywall/로그인 필요 소스 표기

## 도구 사용 패턴

### WebSearch

```
WebSearch("{query}")
```

- 10개 결과 중 도메인 기반 1차 필터 (예: `.gov`, `.edu`, 주요 매체)
- 나머지는 스킵 or aggregator로 분류

### WebFetch

```
WebFetch(url, "Extract: key claims, dates, quoted sources, author affiliation")
```

- prompt는 구체적으로 (전체 요약 X, 핵심 주장·날짜·저자 소속)
- 결과에서 원문 발췌 ≤5문장

### 실패 대응

- `WebFetch` 404/paywall: URL만 기록, "비접근" 표기
- `WebSearch` 결과 빈곤: 쿼리 2~3회 재설계 후 "커버리지 부족" 기록
- 동일 컨텐츠 farm/SEO 팜 감지: "저품질 제외" 사유

## 흔한 함정

### 함정 1: 상위 검색 결과에 SEO 팜

검색 상위가 모두 "Ultimate guide to T" 같은 콘텐츠 팜일 수 있다. 도메인 평판 무시하고 인용하면 원 출처 없이 재해석만 쌓인 보고서가 된다.

**대응:** 각 URL의 도메인·저자·발행일·원 출처 링크를 먼저 본다. 원 출처 링크 없으면 스킵.

### 함정 2: Wikipedia 인용

Wikipedia는 편집될 수 있고 원 출처가 실제와 다르게 요약됐을 수 있다.

**대응:** Wikipedia 페이지의 `References` 섹션에서 원 출처 URL을 찾아 그 1~2차 소스를 인용.

### 함정 3: 자동 생성 콘텐츠

AI가 생성한 모호한 블로그가 상위에 노출되기도.

**대응:** 저자 이름, 프로필, 다른 글을 확인. 저자 정보 없음 + 추상적 문장 일색 → 스킵.

### 함정 4: 시간 표기 누락

"최근", "올해" 같은 상대 표현으로 적힌 글.

**대응:** 발행일을 명시적으로 확인. 없으면 "발행일 확인 불가"로 표기 + 시간 맥락 부속성 낮춤.

## 금지 사항

- 3차 소스를 최종 인용으로 사용
- URL·발행일 누락 인용
- 이해관계 있는 출처를 중립 분석으로 제시
- 추측 기반 "많은 사람들이 말하기를…"
- 학술 논문 영역 침범 (academic-sourcing에 위임)
- 커뮤니티 반응 수집 (community-pulse에 위임)
