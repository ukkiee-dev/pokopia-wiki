# Brief Template

Phase 1에서 리더가 작성하는 `_workspace/datapipeline/{ts}/00_brief.md` 템플릿. 4명 전문가가 이 문서만 읽고 설계를 시작할 수 있어야 한다.

## 템플릿

```markdown
# Pipeline Design Brief

## 1. 도메인·맥락
- 비즈니스 도메인: {예: 전자상거래 주문 분석}
- 주요 이해관계자: {예: BI 팀, 마케팅 분석가}
- 파이프라인 목적: {예: 일일 매출 대시보드 + 주간 코호트 분석}

## 2. 데이터 소스
| 소스 | 종류 | 접근 방식 | 현재 상태 | 비고 |
|------|------|-----------|-----------|------|
| orders-api | REST API | 인증 필요, 페이지네이션 | 운영 중 | rate limit 1000/min |
| customer-db | PostgreSQL | 읽기 리플리카 있음 | 운영 중 | CDC(Debezium) 가능 |
| events-kafka | Kafka topic | 자체 호스팅 | 운영 중 | 보존 7일 |

## 3. 규모 추정
- 일일 유입 레코드 수: {예: 주문 약 500만 건, 이벤트 약 1억 건}
- 일일 데이터 용량: {예: 압축 후 약 30GB}
- 피크 QPS: {예: 주문 200 TPS, 이벤트 5k TPS}
- 과거 데이터: {예: 최근 3년 보존 필요}

**불확실할 때:** 소/중/대 3가지 시나리오로 추정 제공.

## 4. 지연·신선도 요구
- 주요 소비자별 요구:
  - BI 대시보드: 일일 신선도 허용
  - 운영 모니터링: 15분 신선도 필요
  - ML 피처: 1시간 신선도
- 스트리밍 vs 배치 허용 여부: {예: 둘 다 허용, 분 단위 이하는 스트리밍 선호}

## 5. 다운스트림 소비자
- BI/분석: {도구 — Looker, Tableau, Metabase}
- ML: {피처 스토어, 훈련 데이터셋 형식}
- 서비스: {GraphQL API 등}
- Ad-hoc 쿼리: {SQL 허용 대상}

## 6. 운영 환경
- 인프라: {예: AWS, GCP, homelab}
- 선호 스택: {예: dbt + Airflow + Snowflake / 또는 미정}
- 예산 제약: {월 예산 상한 또는 "비용 최소화"}
- 팀 역량: {SQL 전담 / Python 전담 / 양쪽 / 제한적}

## 7. SLA·품질 요구
- 신선도 SLO: {예: p95 < 1시간}
- 완전성 SLO: {예: 일일 99.9% 레코드 적재}
- 정확성 SLO: {예: critical 검증 100%, high 99%}
- 장애 복구 시간: {RTO}
- 데이터 보존: {표준 90일, 감사 7년 등}

## 8. 규제·보안 요구
- PII 존재 여부 + 처리 정책
- 암호화 요구 (전송 / 저장)
- 접근 제어 (열별·행별)
- 감사 로그 요구

## 9. 주요 쿼리 패턴 (상정)
- {쿼리 1: 일별 카테고리별 매출 합계}
- {쿼리 2: 고객 코호트 월별 리텐션}
- {쿼리 3: 특정 상품 상세 페이지 이벤트 시퀀스}

## 10. 가정(Assumptions) — 리더의 기본값
- 사용자가 지정하지 않은 영역에 리더가 채워넣은 값
  - {예: 신선도 목표 미지정 → 4시간 가정}
  - {예: 스택 미지정 → Airflow + dbt + Postgres 가정, 대안 제시}

## 11. 열린 질문
- 사용자에게 확인 필요한 사항 (Phase 1 질의에서 답 못 받은 것)
```

## 수집 우선순위

정보가 불완전할 때 어느 것을 먼저 확보할지:

1. **필수**: 도메인·목적, 데이터 소스 목록, 지연 요구, 주요 소비자
2. **중요**: 규모 추정(적어도 자릿수), 선호 스택(카테고리)
3. **보완**: 예산, 팀 역량, 규제 상세

필수가 불명확하면 Phase 1에서 **3개 이하 질문**으로 사용자에게 확인. 중요·보완은 "가정" 섹션에 기본값 기록하고 진행.

## 규모 시나리오 가이드

사용자 추정이 부재할 때 사용할 기본 시나리오:

| 시나리오 | 일일 행 수 | 일일 용량 | 일반 패턴 |
|----------|-----------|-----------|-----------|
| 소규모 | < 100만 | < 5GB | 배치 충분, 단일 인스턴스 가능 |
| 중규모 | 100만 ~ 1억 | 5GB ~ 500GB | 파티셔닝 필요, 배치 + 선택적 스트리밍 |
| 대규모 | > 1억 | > 500GB | 스트리밍 중심, 세밀한 파티셔닝, 분산 처리 |

전문가들은 이 시나리오 레이블을 공유 어휘로 사용 가능.

## 질의 예시

Phase 1에서 사용자에게 되물을 때 쓰는 포맷:

```
설계를 시작하기 전에 3가지만 확인하겠습니다:

1. 지연 요구: 분석용 일일 배치면 충분한가요, 아니면 분 단위 신선도가 필요한가요?
2. 규모: 일일 유입 레코드가 대략 어느 자릿수인가요? (수만 / 수십만 / 수백만 / 수천만+)
3. 스택 선호: Snowflake/BigQuery 같은 웨어하우스가 이미 있나요, 아니면 일반 권장이 필요한가요?

각 질문에 '모름'이라 답하셔도 됩니다. 그런 경우 기본 가정으로 진행하고 가정 섹션에 명시합니다.
```

질의는 **3개 이하**, "모름" 허용, 각 질문의 **기본 가정**을 명시.
