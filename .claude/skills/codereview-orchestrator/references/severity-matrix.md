# 심각도 매트릭스

모든 감사자가 동일한 기준으로 심각도를 부여하기 위한 표준. 영역별 특화 예시 포함.

## 기본 정의

| 심각도 | 의미 | 대응 기대치 |
|--------|------|------------|
| **critical** | 프로덕션 영향 즉각, 사용자 데이터/서비스 가용성 위협 | 즉시 수정, 릴리즈 블록 |
| **high** | 단기 내 사고 가능, 공격 가능하거나 장애 유발 | 이번 스프린트 내 수정 |
| **medium** | 방치 시 누적되어 문제, 중기 유지보수 부담 | 다음 스프린트 내 수정 |
| **low** | 사소한 개선, 품질 향상에 기여 | 시간 될 때 |
| **info** | 참고 사항, 수정 불필수 | 판단에 맡김 |

## 영역별 심각도 기준

### 아키텍처 (ARCH)

| 심각도 | 기준 | 예시 |
|--------|------|------|
| critical | 시스템이 동작 불능이거나 곧 불능해질 순환/오염 | 런타임 순환 import로 앱 기동 실패 |
| high | 테스트 불가능하거나 주요 기능 추가 시 기존 코드 대거 수정 필요 | 도메인 로직이 3개 이상 인프라에 강결합 |
| medium | SRP/DIP 위반, 리팩토링 비용 중간 | 한 클래스가 2~3개 책임 혼재 |
| low | 미세한 경계 침투, 국소 영향 | 유틸리티 함수의 위치 부적절 |
| info | 개선 여지는 있으나 실질 문제 없음 | 네이밍 컨벤션 예외 1건 |

### 보안 (SEC)

| 심각도 | 기준 | 예시 |
|--------|------|------|
| critical | RCE, 인증 우회, 대규모 데이터 유출 가능 | SQL injection in 공개 API, 하드코딩된 프로덕션 시크릿 |
| high | 권한 상승, 민감 데이터 노출, 공격 난이도 낮음 | XSS, 약한 암호화, IDOR |
| medium | 공격 가능하나 선행 조건 필요, 영향 국소 | 로그에 민감 정보 노출(내부 로그만), CSRF 토큰 누락(GET만) |
| low | 정보 누출 수준, 공격 활용 제한적 | 스택 트레이스 노출, 과도한 에러 메시지 |
| info | 모범 사례 권고 수준 | HSTS 헤더 권장, Content-Security-Policy 강화 |

> **참고:** OWASP Top 10 / CVSS 기준을 참고하되, 프로젝트 맥락(내부 도구 vs 공개 서비스)에 따라 조정.

### 성능 (PERF)

| 심각도 | 기준 | 예시 |
|--------|------|------|
| critical | 프로덕션 장애 유발, 메모리 누수로 시간 내 OOM | 무한 루프 가능 조건, 동기 I/O가 event loop 블록 |
| high | 사용자 요청당 수초 지연 또는 자원 고갈 가속 | 핫패스의 N+1 쿼리, 리스트 n²에서 n=수천 |
| medium | 피크 시 지연, 중기적으로 확장성 제약 | 캐시 가능한 값 반복 계산, 불필요한 JSON.stringify |
| low | 드물게 발생, 사용자 체감 적음 | 관리자 도구의 n² 루프 |
| info | 최적화 기회, 실제 문제는 아직 없음 | Map이 더 적합한 Array 탐색(n<10) |

### 스타일 (STYLE)

| 심각도 | 기준 | 예시 |
|--------|------|------|
| critical | 거의 사용 안 함. 스타일은 보통 critical 아님 | — |
| high | 유지보수가 심각하게 어려운 수준 | 2000줄 god file, 80% `any` 타입 |
| medium | 이해 비용 높음, 버그 유발 가능 | 의미 불명 네이밍, 중첩 깊이 >6 |
| low | 가독성 저해, 국소 개선 | 주석 없는 복잡 정규식, 매직 넘버 |
| info | 집계형 발견, 자동화 권장 | ESLint 규칙 적용으로 일괄 해결 가능 |

## 심각도 판정 체크리스트

판단이 애매할 때 다음을 자문:

1. **악용/사고 가능성은 즉각인가, 조건적인가?** 즉각 → 상향, 조건적 → 하향
2. **영향 범위는?** 전체 사용자/데이터 → 상향, 관리자만/국소 → 하향
3. **수정 비용은?** 작음 → 상향 가치(빨리 고쳐야), 큼 → 우선순위 high로 유지하되 단계적 수정 권장
4. **기존 방어가 존재하는가?** 레이어 방어 있음 → 한 단계 하향 (defense in depth 고려)
5. **테스트로 검증 가능한가?** 불가 → confidence 낮춤 (심각도 하향은 아님)

## 심각도 인플레이션 경계

- critical·high는 "정말로 이번 주 내 조치가 필요한가?"로 재검토
- 1회 리뷰에서 critical이 5개를 넘으면 카테고리 재평가
- info·low는 집계로 처리, 본문에 모두 나열하지 않음

## 공격 가능성 보정 (보안 전용)

`exploit_complexity`를 severity에 반영:

| 기본 심각도 | exploit_complexity | 최종 심각도 |
|------------|-------------------|------------|
| high | low | critical로 상향 고려 |
| medium | low | high로 상향 |
| high | high | high 유지 |
| medium | high | low로 하향 고려 |

## 부하 보정 (성능 전용)

- 핫패스(사용자 요청마다 실행) → 심각도 한 단계 상향
- 배치/관리자 도구 → 심각도 한 단계 하향
- N 크기 불명 → `confidence: medium`, 심각도는 보수적으로
