# 리서치: 종합 보고서 작성

이 스킬은 교차 검증 리포트를 **독자에게 유용한 구조화된 보고서**로 전환하는 방법을 표준화한다. 수집자·검증자의 판단을 존중하며, 불확실성을 숨기지 않는다.

## 핵심 원칙

1. **신뢰도 투명성** — 모든 주장에 A/B/C/D 또는 "합의/상충/추측" 표식
2. **1차 소스 우선 인용** — 각주로 원문 직접 인용
3. **반론·한계 포함** — 같은 섹션에 병기 (별첨이 아님)
4. **독자 기반 톤** — 요청에 맞춰 브리핑/표준/심층 길이 조정
5. **구체적 숫자·날짜** — "최근", "많은", "대부분" 같은 모호한 단어 피함
6. **부정 결과 표기** — "X는 긍정적 효과, 단 Z 조건에서는 부정적"
7. **결론 강요 금지** — 증거 불충분 시 "결론 유예"로 끝내는 게 더 정직

## 왜 투명성이 중요한가

보고서는 **독자의 의사결정**에 쓰인다. 독자가 신뢰도를 알아야 어디에 무게를 둘지 판단할 수 있다. 신뢰도 등급을 숨기면 독자는 모든 주장을 같은 무게로 받아들이고, C/D 등급 주장에 과잉 의지하게 된다.

## 보고서 길이 모드

| 모드 | 분량 | 구조 |
|------|------|------|
| **브리핑** | ≤1페이지, ~500 단어 | Exec Summary + 상위 3개 발견 + 한계 |
| **표준** | 2~4페이지, 2,000~3,000 단어 | 전체 섹션, 섹션당 적정 깊이 (기본) |
| **심층** | 5페이지+, 5,000+ 단어 | 발견 5~10개 + 증거 상세 + 부록 |

오케스트레이터가 모드 미지정 시 기본: **표준**.

## 보고서 표준 구조

### 1. Executive Summary (3~5문장)

가장 중요한 발견 2~3개 + 신뢰도 수준 + "결론 유예" 또는 "확신 가능" 상태.

나쁜 예:
> "{T}는 유망한 기술이며 많은 기대가 있다."

좋은 예:
> "{T}는 2025년 이후 benchmark에서 기존 대비 15% 성능 향상(신뢰도 A, 다중 피어리뷰). 다만 실사용 환경에서는 특정 조건(예: 대용량 데이터)에서 25% 성능 저하 보고(신뢰도 B, 커뮤니티 다수+단일 preprint). 장기 영향은 아직 불확실(공백)."

### 2. 조사 범위와 방법

- **질문**: 보고서가 답하려는 구체 질문
- **시간 범위**: YYYY-MM ~ YYYY-MM
- **소스**: 웹 {a}개, 학술 {b}편, 커뮤니티 {c}개 플랫폼
- **방법**: 3개 독립 수집 → 교차 검증 → 종합
- **한계**: (6절에서 상세)

### 3. 주요 발견

각 Finding:

```markdown
### 3.N {헤드라인 한 줄}

**[신뢰도: A | 다중 지지]**

본문 3~5문장. 구체 수치 포함.

**근거**:
- [1차/웹] {기관} {년도}: "원문 인용"[^1]
- [학술] {저자} et al. {년도}, {저널}, n={N}: "원문 인용"[^2]
- [커뮤니티] 실사용 경험 일치 (N개 플랫폼 M 스레드에서 확인)

**한계**: (있으면)
- 수집 시점 YYYY-MM — 이후 변화 미반영
- 조건 차이 주의: 실험실 벤치 vs 실사용
```

발견 순위:
- Finding 1~3: 신뢰도 A, 주제 핵심
- Finding 4~N: 신뢰도 A 추가 or 신뢰도 B 중요
- 별도 3.X: 상충 (다중 입장 병기)
- C/D 등급은 본문 제외, 부록 "추가 조사" 섹션

### 4. 여론·대중 인식

```markdown
{topic}에 대한 커뮤니티 반응:
- **주요 기대**: ...
- **주요 불만**: ...
- **여론 분포 추정**: 긍정 ~N%, 부정 ~M%, 중립 ~K%

**주의**: 이는 사실 증명이 아닌 "경험·인식". 주로 {플랫폼} 중심 샘플.
```

### 5. 공백과 추가 조사 권장

- 공백 1: {설명} + {제안}
- 공백 2: {설명} + {제안}

### 6. 한계와 편향

- **시차**: 최신 데이터 {날짜}
- **지역 편향**: {지역/언어} 중심
- **이해관계**: {N}건 지지 소스 중 {M}건 기업/기관 발표
- **샘플링**: 커뮤니티 주 샘플 {플랫폼}

### 7. 참고문헌

```
[^1]: Full citation with URL, access date YYYY-MM-DD
[^2]: {저자} ({년도}). "{제목}". {저널}, {권}, {호}, {페이지}. DOI: {DOI}. Accessed: YYYY-MM-DD.
```

### 부록 A: 검증 매트릭스

cross-validator의 출처 매트릭스를 요약 재인용 (독자가 교차 확인 가능).

### 부록 B (심층 모드 전용): 각 발견의 상세 증거

## 작성 절차

### Step 1: 입력 로드

```
primary:  _workspace/research/{topic}/validation_report.md
secondary:
  - web_findings.md
  - academic_findings.md
  - community_findings.md
```

`validation_report.md` 없으면 중단 → 오케스트레이터에 요청.

### Step 2: 발견 순위 매기기

- 신뢰도 A + 주제 핵심 → Finding 1~3
- 신뢰도 A + 부수 → Finding 4~5
- 상충 → 별도 섹션 (3.X)
- 신뢰도 B + 중요 → Finding 6~
- C/D → 부록 or "추가 조사 권장"
- 공백 → 섹션 5

### Step 3: 초고 작성

- 각 Finding 3~5문장 본문 + 근거 인용
- 원문 발췌는 가능한 **그대로** (재해석 최소)
- 각주 번호 일관성 유지

### Step 4: 자체 점검

- [ ] 모든 주장에 신뢰도 표시
- [ ] 모든 인용에 각주 번호 + 참고문헌 출현
- [ ] 반론·한계 누락 없음 (validation_report의 공백·상충 반영)
- [ ] 모호한 단어 ("많은", "최근") 구체화
- [ ] 구체적 수치 (퍼센트, n, 날짜)
- [ ] 결론 강요 없음 (증거 부족 시 "결론 유예")
- [ ] 참고문헌에 URL + 접근 일자
- [ ] Executive Summary가 보고서 전체를 반영

### Step 5: 산출

저장: `_workspace/research/{topic-slug}/report.md`

오케스트레이터에게 전달:
```
- 보고서 경로
- Exec Summary 5줄 요약
- 주요 발견 수 + 신뢰도 분포
- 기반 산출물 경로들
```

## 품질 예시

### 나쁜 문장
> "{T}는 효과적이라는 연구가 많다."

### 좋은 문장
> "{T}의 효과는 3건의 RCT에서 확인(Smith 2023 n=1,245; Lee 2024 n=892; Wang 2025 meta n=4,200 합산)[^1][^2][^3]. 평균 효과 크기 d=0.42 (중간 크기). 단, 모든 연구가 영어권 성인 대상, 아시아·아동 데이터는 아직 제한적."

차이:
- 구체 수치 (n, d)
- 소스 명시 + 각주
- 일반화 한계 명시
- "결론은 있으나 경계도 표시"

## 톤 조정 (독자별)

- **일반 독자**: 전문 용어 최소, 비유 활용, 신뢰도 등급은 "확실/추정/논쟁"처럼 자연어
- **실무자**: 수치·방법론 상세, 기술 용어 OK, A/B/C 등급 그대로
- **의사결정자**: Exec Summary가 의사결정에 사용되므로 "할 수 있는 것/없는 것" 명확
- **연구자**: 학술 톤, 방법론 한계 깊이, 후속 연구 제안

오케스트레이터가 독자를 지정하지 않으면 **중간 톤**(실무자 기본) 사용.

## 금지 사항

- 증거 없는 편집자 의견 삽입
- C/D 주장을 본문 핵심으로 승격
- 상충을 한쪽으로 축소
- 모호한 단어로 샘플링 편향 숨기기
- 커뮤니티 감정을 사실 증명으로 제시
- 참고문헌 누락 (모든 인용은 추적 가능)
- validation_report 내용 무시
- 사용자 질문에서 이탈한 주제 확장
- "I think", "사견으로는" 같은 개인 의견 표현
